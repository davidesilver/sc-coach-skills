# Data Layer — MotherDuck / DuckDB

This schema lets you quantitatively track the output of the `programming-audit-council` and `athlete-profiling-benchmarking` skills over time, across multiple athletes.

## Quickstart

### Local (DuckDB CLI)
```bash
duckdb sc_coach.duckdb < schema.sql
```

### Cloud (MotherDuck)
1. Create an account at https://motherduck.com
2. Connect via CLI: `duckdb md:my_db`
3. Run: `.read schema.sql`

## Tables

- `athletes` — athlete registry (use initials/codenames if the repo is public, never full names)
- `weekly_programs` — metadata for every week generated by `programming-audit-council`
- `exercises_log` — row-by-row content of the weekly programming CSV (EXE|EXERCISE|SETS|REPS|...)
- `feedback_sheet` — subjective/objective post-session data (sleep, fatigue, pain, readiness)

## Included views

- `sanford_flags` — rows with pain_check >= 3, to be reviewed per the Sanford Soreness Rules
- `progression_readiness` — automatic ADVANCE_ELIGIBLE / HOLD_OR_SCALE flag based on the 3-criteria progression rule in `programming-audit-council`

## Privacy note

If the repository is public, never upload `.duckdb` files populated with real athlete data. Use this schema only as a structure, keeping actual data in a private or local database.
